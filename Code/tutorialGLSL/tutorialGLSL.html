GLSL Tutorial

The modern graphics pipeline Tutorial

Recent versions of OpenGL offer a lot more flexibility to the programmer for controlling rendering and effects like lighting themselves. They no longer include built-in commands like glLight and glTranslate, instead you have to implement the matrix math and lighting calculations yourself. This means that to simply render a triangle, we need to write a little bit lot more boilerplate code, but it allows you to implement much more interesting effects.

In this tutorial, we will guide you through every step to implement a basic graphics pipeline in modern OpenGL. The result is a program that renders a 3D Model to the screen, where each pixel is shaded according to the normal of the face it corresponds to. While this isnt particularly useful, it serves as a good starting point for you to implement your own more interesting graphics effects.

In Modern OpenGL you need to implement a custom version of the graphics pipeline by writing shaders. These are small programs written in a domain specific programming language optimized for implementing a specific stage of the graphics pipeline on the GPU:

The simplified graphics pipeline, for rasterizing a triangle to the screen consists of 3 steps:

  1. The 3D vertex positions are projected to 2D vertices on the screen (perspective projection math)
  2. Rasterization takes place, which computes which pixels in the screen correspond to the projected triangle
  3. For each pixel in the screen belonging to a triangle, we need assign a color to it.

Step 2 happens automatically on highly optimized hardware in the GPU, but steps 1 and 2 need to be implemented by the programmer in the form of vertex and fragment shaders respectively.

Vertex Shaders

As we mentioned before, the vertex shader is a program responsible for mapping the 3D coordinates of our model to 2D positions in the screen. An example of a simple vertex shader looks like this:

#version 410

// MVP matrix passed as uniform
layout uniform mat4 mvp;

// vertex position passed as input
layout in vec3 pos;

void main() {
    // output is the transformed vertex according to the MVP matrix
    gl_Position = mvp * vec4(pos, 1.0);
}

As you can see, shaders are written in a C-like programming language, which is called GLSL. In general, a vertex shader receives as input a set of vertex attributes, and outputs a transformed set of vertex attributes of the same length. It is executed independently and in parallel for each vertex on the GPU.

This shader takes the 3D position of the point and multiplies it with the combined model/view/projection matrix to transform it into a position on the screen. The gl_Position variable is a special output variable in OpenGL shaders that the program has to write its screen position to.

To calculate this position, you can use several types of variables in your shader. Data that is the same across all of your points, like the perspective matrix, should be passed as a global variable, known as a uniform in OpenGL shaders. You can set these values from your C++ program. Because the vertex shader runs for every point in your model i.e. every vertex, it also receives per-vertex data like the position. These per-vertex values are passed as in variables.

Fragment Shaders

After points in your model have been transformed into screen coordinates by the vertex shader, the graphics card will combine them into points, lines or triangles. This process is known as rasterization and determines which pixels on the screen every triangle, for example, will cover.

To determine the color of each pixel in a triangle, a second type of shader is run called the fragment shader. This is a shader program that runs for every pixel inside shapes that are drawn and should output the final color that you see on the screen. A simple fragment shader looks like the following code.

#version 410

out vec4 color;

void main() {
    color = vec4(1.0, 0.0, 0.0, 1.0);
}

This fragment shader will color every pixel of the shapes you draw red. Just like the vertex shader, you have to write your final result to a variable. In the case of fragment shaders, that is a special out variable. The reason that this is a variable that you have to define yourself instead of there just be a gl_Color is that it is possible to output multiple colors from a fragment shader, but this is advanced functionality.

Implementing our graphics pipeline

By now you should have a general understanding of how vertex and fragment shaders work. To summarize this is what we’ve learned so far:

  1. You specify per-point data for your model, like positions, normals and texture coordinates.

  2. The vertex shader is run for each point with these attributes as input and it will output an on-screen position, and possibly other attributes for all of them.

  3. The graphics card takes the on-screen points and combines them into shapes. For triangles, for example, it will create groups of 3. It will determine which pixels on the screen are covered by that triangle and then it will run the fragment shader for each of them to determine the color to put on the screen.

Now lets go through a concrete example, that shows us how to render a 3D Triangle mesh in C++ using modern OpenGL. After loading a 3D model in C++ we usually have our mesh in a format that looks something like this:

struct Vertex {
    glm::vec3 color;
    glm::vec3 pos;
};

struct Mesh {
    // vertices contain vertex positions and normals of the mesh
    std::vector<Vertex> vertices;

    // each entry in the list represents a triangle and is a is vector of unsigned
    // integer triplets corresponding to the indices in `vertices` of the 3 vertices that make up the triangle
    std::vector<glm::uvec3> triangles;
}

Passing vertex data to the shaders (vertex and index buffers)

The first step is to send this data to the GPU so that the vertex shader can use it as input. To send the vertex data to the GPU, we allocate a buffer on the GPU and put this data on it. By convention, we usually call the buffer storing the vertex data the Vertex Buffer Object (VBO). To create a buffer in the GPU we use glGenBuffers, glBindBuffer and glBufferData, :

GLuint vbo;
glGenBuffers(1, &vbo);
glBindBuffer(GL_ARRAY_BUFFER, vbo);
glBufferData(GL_ARRAY_BUFFER, static_cast<GLsizeiptr>(mesh.vertices.size() * sizeof(Vertex)), mesh.vertices.data(), GL_STATIC_DRAW);
glBindBuffer(GL_ARRAY_BUFFER, 0);

Likewise, we need to send the triangle data to the GPU. We do this with another buffer, which is usually called the Index Buffer Object (IBO):

GLuint ibo;
glGenBuffers(1, &ibo);
glBindBuffer(GL_ELEMENT_ARRAY_BUFFER, ibo);
glBufferData(GL_ELEMENT_ARRAY_BUFFER, static_cast<GLsizeiptr>(mesh.triangles.size() * sizeof(decltype(Mesh::triangles)::value_type)), mesh.triangles.data(), GL_STATIC_DRAW);
glBindBuffer(GL_ELEMENT_ARRAY_BUFFER, 0);

Keep in mind that we are using OpenGL 4.1 and all OpenGL calls in this version are stateful. Each time we modify a buffer (or texture), we bind a buffer with glBindBuffer, which OpenGL remembers untill we unbind it by binding buffer 0.

Writing the Vertex and Fragment shaders

We can now write our vertex shader. To compute the 2D position on the screen for each vertex we use the mvp matrix, which is passed as a uniform by multipliying each position with it. Additionally, we can output additional vertex attributes to pass to the fragment shader, in this case since we want to color according to the normal, we declare an output variable to the shader called fragColor and set it to the normal:

#version 410

// read mvp matrix as uniform
layout uniform mat4 mvp;

// position and normal as input vertex attributes
layout in vec3 pos;
layout in vec3 normal;

// pass the normal as an output vertex attribute
out vec3 fragNormal;

void main() {

    // set gl_position to 3D pos transfomred according to mvp
    gl_Position = mvp * vec4(pos, 1.0);

    // set fragColor to the normal
    fragNnormal = normal;
}

Next we write our fragment shader. The fragment shader runs for each pixel, and takes as input the interpoalted output vertex attributes of the triangle it corresponds to. In our case, since we only have one additional output attribute in the vertex shader, our fragment shader takes as input our fragNormal, and outputs color by simply writing the normal to it:

#version 410

// output RGBA color
layout out vec4 color;

// input is output from vertex shader
in vec3 fragNormal;

void main() {
    // use normal as output color, with alpha channel set to 1
    color = vec4(fragNormal, 1.0);
}

Using the framework provided in the exercises, we can then construct our Shader object in C++ by reading the files storing the fragment and vertex shader:

// Construct Shader Pipeline
const Shader debugShader = ShaderBuilder()
    .addStage(GL_VERTEX_SHADER, "shaders/vertex.glsl")
    .addStage(GL_FRAGMENT_SHADER, "shaders/debug_frag.glsl")
    .build();

How to interpret the VBO and IBO: Vertex Array Object

There’s still one piece missing from the puzzle. When we created our vertex and index buffers we simply passed some arrays array as binary data, but htese arrays could use any kind of C++ struct with variable sizes, names, orders, etc, so OpenGL doesnt know how to interpret the buffers.

To solve this, you additionally need to give OpenGL a mapping between the values in your vertex array and the inputs to your shader. To accomplish this, OpenGL has a special Vertex Array Object that can store these mappings for you. Don’t mind the misleading name, it stores mappings and not vertex arrays. To link the aforementioned vertex buffer with positions and colors to the vertex shader, you would use the following code:

 GLuint vao;

// Create VAO and bind it so subsequent creations of VBO and IBO are bound to this VAO
glGenVertexArrays(1, &vao);
glBindVertexArray(vao);
glBindBuffer(GL_ARRAY_BUFFER, vbo);
glBindBuffer(GL_ELEMENT_ARRAY_BUFFER, ibo);

// The position and normal vectors should be retrieved from the specified Vertex Buffer Object.
// The stride is the distance in bytes between vertices. We use the offset to point to the normals
// instead of the positions.
// Tell OpenGL that we will be using vertex attributes 0, 1 and 2.
glEnableVertexAttribArray(0);
glEnableVertexAttribArray(1);

// We tell OpenGL what each vertex looks like and how they are mapped to the shader
glVertexAttribPointer(debugShader.getAttributeLocation("pos"), 3, GL_FLOAT, GL_FALSE, sizeof(Vertex), (void*)offsetof(Vertex, position));
glVertexAttribPointer(debugShader.getAttributeLocation("normal"), 3, GL_FLOAT, GL_FALSE, sizeof(Vertex), (void*)offsetof(Vertex, normal));

glBindVertexArray(0);

We first tell the Vertex Array Object which element buffer to use. Secondly, we specify the Vertex Buffer Object, the index of the shader input and how it is laid out in memory. Shader inputs are disabled by default, so they need to be enabled using glEnableVertexArrayAttrib by their index.

NOTE: We are using OpenGL 4.1 released in 2010, which is the latest version Apple supports. This version does not support layout qualifiers, meaning uniforms, and vertex attributes locations can change between shaders. This means you might either need multiple VAOs, or otherwise not storing the vertex atribute pointers in the VAO and define them separetly every time (This is our approach in exercise 3).

Passing Uniforms to shaders

You now know how per-vertex data is provided to shaders to draw geometry on the screen. However, you also need to provide some data for your global (uniform) shader variables. Specifically in our example we need to provide the mvp object to the vertex shader, this is done by index with the glUniform functions, in our case we are passing a 4x4 float matrix so we use glUniformMatrix4fv:

// compute the mvp matrix
const glm::mat4 view = trackball.worldToCameraMatrix();
const glm::mat4 proj = glm::perspective(fieldOfView, aspectRatio, clipNear, clipFar);
const glm::mat4 model = glm::mat4(1.0f); 
const glm::mat4 mvp = proj * view * model;

// pass the mvp matrix as a uniform
glUniformMatrix4fv(
    debugShader.getUniformLocation("mvp"),
    1,
    GL_FALSE,
    glm::value_ptr(mvp)
);

Shader uniforms can be set directly with functions like glUniformMatrix4fv, you don’t have to explicitly put them in graphics card memory like vertex data. In this code snippet we use the GLM library to set up a view, projection and model transformation. These matrices are multiplied to create a combined model/view/projection matrix that is then uploaded to the shader uniform variable called mvp (see the vertex shader code above). This code is slightly different from what you might have seen in the lectures. We need to use OpenGL 4.1 to support Apple computers, this version did not yet have explicit layout definition and thus we need to let the driver decide uniform locations while compiling the shader. Getting the binding point is done using the provided Shader::getUniformLocation(const char*) function. Internally it uses the OpenGl glGetUniformLocation call.
